# 项目背景

## 项目概述

随着大语言模型的发展，游戏NPC已经具备自然语言交互能力，但现阶段AI NPC仍然本质上是“被动响应式聊天机器人”。

它们能够回答问题，却无法真正理解自己是谁、身处什么环境、经历过什么事情，以及未来应该如何成长。

玩家获得的是一次次独立的对话，而不是与一个持续存在的角色共同生活。

本项目希望构建一套基于人格、记忆、世界和时间驱动的NPC生命系统，使NPC能够在虚拟世界中持续成长、经历事件、积累记忆并形成稳定人格，从而实现更接近真实生命体的长期互动体验。

------

## 行业现状

当前主流AI NPC方案主要采用：

```text
玩家输入
↓
Prompt组装
↓
LLM生成
↓
角色回复
```

虽然提升了语言自然度，但角色本身仍存在严重缺陷。

无论是游戏NPC还是AI陪伴角色，其智能主要集中于“说什么”，而非“为什么这样说”。

角色缺少对自身、世界和时间的理解，因此无法形成持续成长能力。

------

# 当前痛点分析

## 痛点一：角色世界观不稳定

现有AI角色通常只保存设定文本。

例如：

```text
你是一名酒馆老板
```

但模型并不真正理解：

- 酒馆在哪里
- 城镇是什么样
- 最近发生什么事件
- 自己与其他角色关系如何

因此会出现：

```text
昨天住在王城
今天住在边境
明天又来自海岛
```

角色认知经常发生漂移。

### 本质问题

缺少持续运行的世界模型（World Model）。

------

## 痛点二：角色无法感知当前场景

大多数NPC对话脱离环境。

例如：

玩家站在战场：

```text
村庄已经被攻陷
到处都是尸体
```

NPC却回答：

```text
今天天气真不错
```

因为模型只看到了用户输入。

没有看到：

```text
当前地点
当前时间
周围人物
正在发生的事件
```

### 本质问题

缺少实时场景系统（Scene System）。

------

## 痛点三：角色不会主动推动剧情

现有NPC主要依赖玩家驱动。

玩家不开口：

```text
NPC什么都不会发生
```

导致：

- 世界静止
- 角色静止
- 剧情静止

NPC缺少目标与计划。

### 本质问题

缺少行为驱动系统（Behavior System）。

------

## 痛点四：角色没有自己的生活

玩家离线后：

```text
NPC暂停存在
```

再次上线：

```text
NPC仍然停留在原地
```

现实中的人会：

- 工作
- 睡觉
- 社交
- 产生新经历

而AI NPC不会。

### 本质问题

缺少后台生命模拟系统（Life Simulation）。

------

## 痛点五：记忆缺乏时间组织

当前大部分记忆系统：

```text
向量数据库
+
相似度检索
```

存在问题：

### 时间顺序混乱

NPC可能同时回忆：

```text
三个月前的事情
昨天的事情
十分钟前的事情
```

无法判断先后关系。

------

### 记忆重要性混乱

NPC可能记住：

```text
你喜欢吃苹果
```

却忘记：

```text
你救过他的命
```

------

### 记忆激活错误

在当前情境下不相关的记忆被频繁召回。

导致角色行为异常。

### 本质问题

缺少时间驱动的记忆编排系统（Temporal Memory System）。

------

## 痛点六：人格不会成长

当前人格设定：

```text
勇敢
善良
乐观
```

从创建到删除始终不变。

而真实人格会受到：

- 时间
- 经历
- 关系
- 创伤
- 成功

影响。

例如：

```text
乐观
↓
战争经历
↓
谨慎
↓
背叛经历
↓
冷漠
```

人格应当是动态演化的。

### 本质问题

缺少人格演化系统（Personality Evolution）。

------

# 用户分析

## 核心用户

### RPG玩家

需求：

- 角色拥有成长轨迹
- 剧情能够持续发展
- 关系长期演变

他们期待：

> 与一个活着的人互动，而不是聊天窗口。

------

### 开放世界玩家

需求：

- 世界持续运行
- NPC拥有自己的生活

他们期待：

> 即使我离线，世界仍在运转。

------

### AI陪伴用户

需求：

- 长期陪伴
- 共同经历
- 情感积累

他们期待：

> 角色能够和自己一起成长。

------

# 产品定位

## 产品定义

> 一套基于人格、世界、时间和记忆驱动的NPC生命系统，为游戏角色提供持续存在、自主成长和长期演化能力。

------

## 核心定位

不是：

```text
AI聊天系统
```

也不是：

```text
AI NPC系统
```

```text
NPC生命模拟系统
```

# 后续版本实现计划

> MVP v1.0 已实现核心闭环：创建 → 对话 → 成长 → 状态查看。以下为后续版本迭代方向。

---

## 一、立即优化（v1.1）

针对 MVP 版本中已知体验问题和未完善的交互细节，进行集中优化。

### 1. 修复前端"当前状态"栏文本过大显示不全问题

**现状问题**：
- 角色创建页和状态面板页的"当前状态"区域使用 `st.metric` 展示 location / activity / mood 三个字段
- 当 Creation LLM 生成较长的位置名称或活动描述时（如"王国北境风雪山脉脚下的旧图书馆二楼"），`st.metric` 组件无法自动换行，导致文本被截断或溢出
- 状态面板页的右侧布局空间有限，进一步加剧显示不全

**优化方案**：
- 将 `st.metric` 替换为 `st.markdown` 或 `st.caption` 多行展示，确保长文本自动换行
- 在 CSS 层面添加 `word-break: break-word` 和 `overflow-wrap: break-word` 样式
- 对于超长位置名称，在前端截断并添加悬停 tooltip 展示完整内容

### 2. 加入导演层（Director）输出显示

**现状问题**：
- 对话交互页目前仅在展开的调试面板中显示 Director/Actor 原始 JSON，普通用户不易发现
- Director 输出的 `goal`（对话目标）、`style`（回复风格）、`focus_memories`（激活的记忆）等关键信息未在对话气泡中呈现
- 这导致系统的"可解释性"设计理念未充分传递给体验者

**优化方案**：
- 在每条 NPC 回复气泡下方，默认展示 Director 关键输出摘要（以轻量标签形式）：
  - 目标：`goal` 字段
  - 风格：`style` 字段
  - 关注记忆数：`len(focus_memories)` 条
- 使用 `st.caption` + 浅色背景标签样式，不与 Actor 输出标签（情绪/动作/表情）混淆
- 可折叠展开查看完整的 `focus_memories` 列表内容
- 保留现有调试面板中的完整原始 JSON 输出

### 3. 体现当前场景与上下文提示（to user）

**现状问题**：
- 用户在对话页只能看到角色名称和人格摘要，缺少对角色"当前所处的时空背景"的直观感知
- 例如用户不清楚角色现在在"书咖"还是"家里"、正在做什么（读书/工作/休息）
- 缺少场景提示导致用户不知道该说什么、从什么角度切入对话

**优化方案**：
- 在对话页顶部角色信息栏中，增加一个**场景上下文卡片**（`st.info` 或自定义容器）：
  ```
  📍 现在你在：林雨晴的书咖「晴·雨」
  🕐 时间：一个下雨的午后
  🔄 角色正在：泡一杯茉莉花茶，团子在窗边打盹
  ```
- 场景信息来源：
  - `current_state.location` + `world_setting` 拼合环境描述
  - `current_state.activity` + `current_state.mood` 构成角色即时状态
- 给出一个对用户的**轻度引导提示**（如"你可以跟她聊聊天气、书、或者她的猫"），降低用户输入门槛

### 4. 支持从文件创建角色时输入寄语

**现状问题**：
- 文件上传创建角色时，后端仅将 TXT 文件内容作为角色故事描述传给 Creation LLM
- 用户无法在文件中额外附加自己对角色的期望寄语（如"我希望她是一个有忧郁气质但内心坚强的角色"）
- 文本描述模式下用户可以在 text_area 中混合故事+寄语，但文件模式完全依赖 TXT 内容本身

**优化方案**：
- 在文件上传模式下，新增一个可选的文本输入框"💬 创建寄语（可选）"
- 寄语内容将拼接到 TXT 文件内容末尾，作为 Creation LLM 的额外约束：
  ```
  {文件故事内容}
  
  [额外的角色期望]
  {用户寄语}
  ```
- 后端 `POST /api/characters/create` 接口已有的 `description` Form 字段可复用为此用途，前端将寄语值传入即可

## 二、计划实现（v1.2+）

以下为中长期迭代方向，着眼于系统的完整性和多角色互动能力。

### 1. 同一世界下角色统一管理与联动事件

**目标**：多个角色存在于同一个世界观中，能够相互感知和互动。

**计划内容**：
- **世界模型实体化**：将 `world_setting` 从角色的文本字段提升为独立的 `World` 实体表，包含世界名称、地理信息、时间线、事件列表
- **角色归属**：角色创建时可选加入已有世界，或创建新世界。世界观共享同一时间线
- **角色间关系表**：新增 `Relationship` 表，记录两个角色之间的关系类型（朋友/敌人/师徒/恋人等）和关系值（0–100）
- **跨角色联动事件**：Growth 系统在分析时，若同一世界下有其他角色，可生成"与 XX 角色发生的交互事件"，影响双方人格和关系值
- **世界时间推进**：统一的 `WorldTime` 管理，一天结束后所有角色统一触发成长

### 2. 扩充人格系统字段

**目标**：当前六维人格模型较为精简，需增加更多心理维度以刻画更丰富的角色人格。

**计划内容**：
- **新增维度候选**：
  - 好奇心（Curiosity）：对新事物的探索意愿
  - 果断（Decisiveness）：做决定的速度和坚定程度
  - 宽容（Forgiveness）：原谅他人过错的意愿
  - 自律（Discipline）：自我控制和坚持能力
  - 浪漫（Romance）：对情感和美的敏感度
- **人格分组**：将维度分组（如情感类、行动类、社会类），在状态面板中分 tab 展示，避免信息过载
- **人格演化规则**：为人格变化添加约束规则，如"重大创伤事件对乐观的影响幅度应大于日常对话"，使演化更合理
- **人格兼容性**：未来两个角色互动时，基于人格向量计算兼容度（余弦相似度），影响互动风格和关系变化方向

### 3. 支持用户发送事件

**目标**：用户不仅可以与角色对话，还可以主动触发"事件"来推动角色生活。

**计划内容**：
- **事件输入界面**：在对话页增加"📅 发送事件"按钮，弹出事件描述输入框。用户描述一件发生的事情（如"镇上来了一位神秘商人"、"今天下了一场暴风雪"）
- **事件处理管线**：后端新增 `EventModule`，处理用户事件：
  1. 将事件内容作为特殊上下文注入到后续对话中
  2. 事件自动生成关联记忆（备忘录类型为 `event`）
  3. 事件影响角色的 `current_state`（如"暴风雪" → location 或 activity 变化）
- **事件与成长联动**：用户事件被纳入 Growth 分析输入，影响人格演化方向
- **事件链**：同一角色的事件按时间线排列，在状态面板中可视化展示事件时间轴

### 4. 多模态输入输出

**目标**：突破纯文本交互限制，支持语音、形象等多模态交互。

**计划内容**：
- **语音输入**：前端集成 Web Speech API 或 Whiper API，支持用户语音输入转为文本
- **语音输出（TTS）**：角色回复通过 TTS（如 Edge-TTS / ElevenLabs）转为语音播放，可配置不同声线匹配角色性格
- **虚拟形象展示**：
  - 基于角色人格和描述，调用图像生成模型（如 Stable Diffusion / DALL·E）生成角色立绘
  - 对话时根据 emotion 字段动态切换表情（开心/悲伤/愤怒/惊讶等），实现简单的 Live2D 效果
- **场景可视化**：基于 `world_setting` 和 `current_state.location`，生成当前场景的背景图，增强沉浸感
- **表情适配**：Actor 输出的 `expression` 字段映射到面部表情参数，驱动虚拟形象的微表情变化
